# RSSwithAI 后台管理系统功能与设计文档

## 1. 系统概述与全局架构

本项目（RSSwithAI）的后台管理前端是一个基于 Vue 3 + TypeScript 开发的单页面应用（SPA）。整个系统采用了经典的“侧边栏 + 顶部导航条 + 主内容区”的布局设计。
主要功能模块包括：认证登录、RSS源管理、文章查看与AI增强、模型配置、提示词模板管理、AI分析实验管理以及系统全局配置。面向的主要是系统管理员以及AI实验分析人员。

### 1.1 全局路由与权限控制
- 路由基于 `vue-router` 实现，采用 WebHistory 模式。
- **权限拦截**：系统具备全局路由守卫（Router BeforeEach 拦截器），通过检查本地 token（由 `@/utils/auth` 管理）来控制访问权限。除了 `/login` 页面允许匿名访问外，其它所有涉及内部管理的路由均受到保护。已登录用户如果访问 `/login`，会被自动重定向回内部首页。
- **默认落地页**：登录成功后的默认落地页是 `/rss-sources`（RSS源管理）。

### 1.2 骨架布局结构 (MainLayout)
系统级骨架（`src/layouts/MainLayout.vue`）主要包含三个区域：
- **可折叠侧边栏导航**：提供系统的各大核心模块入口，支持展开与折叠（宽度在 64/20 Tailwind 单元之间切换）。包含的菜单项有：RSS源管理、文章管理、模型配置、提示词模板、实验管理、系统配置。当前激活的路由项会有明显的蓝色高亮背景。
- **顶部栏**：提供当前页面的标题展示，右侧提供全局的“退出登录”按钮（清空 token 并跳转回登录页）。
- **主内容区**：占据界面的主要空间，通过 `<router-view>` 动态挂载与当前路由匹配的具体页面内容页面，且支持内部垂直滚动。

## 2. 详细功能模块说明

### 2.1 登录模块 (Login)
**页面路由**：`/login`

**1. 界面展示**
- 采用了居中的登录卡片设计，背景有轻微的蓝紫色渐变效果。
- 包含系统 Logo（RSSwithAI）以及用户名、密码输入框。
- 登录按钮在接口请求期间会改为“登录中...”的提示并处于禁用状态，防止重复提交。

**2. 核心交互与逻辑实现**
- 数据校验：前端首先校验用户名和密码是否为空，为空时直接阻止提交并予以文字提示。
- 请求与状态管理：
  - 成功：调用 `login` API，获取 token 后通过 `setToken` 保存到浏览器 `localStorage`，然后路由跳转至首页 `/rss-sources`。
  - 失败：捕获报错信息，提取并显示在输入框下侧的红色警示框中。


### 2.2 RSS源管理模块 (RSS Sources)
**页面路由**：`/rss-sources`

**1. 界面视图结构**
页面主要从上到下分为三个维度进行展示：
- **顶部统计卡片**：展示 5 个核心指标：RSS源总数、成功抓取数、抓取失败数、从未抓取数和正在抓取数。指标通过 `StatCard` 组件带颜色呈现。
- **中间操作栏**：包含“新增RSS源”、“抓取全部”的左侧操作组和右侧的“刷新列表”按钮。
- **底部数据列表**：分页展示所有 RSS源 的主体信息（如 ID、名称、可悬浮截断的URL、类型、抓取间隔、系统状态和最终抓取状态标签等），且右侧为操作空间（详情、编辑、删除、启停用、单源手动抓取）。

**2. 核心交互与功能实现逻辑**
- **数据加载与分页**：页面在 `onMounted` 钩子或主动点击刷新时，会并发调用获取统计数据和列表数据的接口。底层依赖独立封装的 `Pagination` 组件维护翻页状态。
- **新增与编辑RSS源**：
  - 复用同一个 `Dialog` 弹窗。配置表单包含名称（如果不填系统会在抓取时尝试从源中级联提取补全）、源URL（核心必须配置）、原始网页链接、源类型（选项为原始RSS源与RSSHub源）、内容分类、描述说明以及抓取频率（默认 30 分钟）。
  - 根据 `dialogMode` 区分是创建行为还是修改覆盖行为，保存完成后重载列表页。
- **源详情查看**：
  - 点击列表详情会专门请求一次单条详情 API，以获取该源的完整状态（例如系统抓取的累计失败次数 `failureCount`，或是最近一次抛出的详细堆栈文本信息 `lastFetchError` ），在独立的视图只读弹窗中进行呈现。
- **源管理控制**：
  - 启/停用开关将立刻触发后端的针对性状态突变。
  - 全部抓取/单击抓取为触发后台的异步消费队列服务，浏览器前端会通过 `alert` 给予触发成功的轻量级反馈。


### 2.3 文章管理模块 (Articles)
**页面路由**：`/articles` (列表页) 和 `/articles/:id` (详情页)

#### 2.3.1 文章列表页
**1. 界面视图结构**
- **顶部统计与趋势池**：左侧展示系统文章总数，右侧挂载一个通过 `Chart.js` 渲染的折线图，直观展示“最近7天每日新增文章数量的趋势”。
- **筛选与检索栏**：支持按“RSS源”进行维度切分（下拉选择），也支持按“搜索关键词”去模糊匹配标题和作者等长尾字段。
- **数据列表**：按时间倒序等业务规则呈现文章的基线项标识集（ID、来源站点、高亮截断标题、作者、发布时间和分类映射）。整行具备超链接属性。

**2. 核心交互与功能实现逻辑**
- **路由状态感知持久化**：在跳转去下钻子页面（文章详情页）时，前端设计了路由携带保护，会将当前的所有查询分片结构（如：选择的 sourceId、输入了一半的 searchWord、当前所在的分页页码 page 与大小 size）压入 URL Query 中。保证分析人员在分析完退回列表时，其复杂组合过滤视图能原样重新渲染激活（`onMounted` 钩子读取加载）。

#### 2.3.2 文章详情页
**1. 界面布局（双栏结构）**
提供左侧原生文章、右侧增强分析的对照查阅形态：
- **左栏（原生文章内容）**：
  - 元信息展示：标题、来源、发布时间等静态属性集萃定。
  - 正文视图切换：针对采集到的文章 `content`，前端注入了富文本容错处理环节。其直接内置了基于 `marked.js` 的 Markdown 到 HTML 的动态降级渲染与 `highlight.js` 的代码染色体支持。并且通过统一注入 `url-rewrites` 进行外部图像在控制台的代理显示（防止防盗链无法加载），同时也提供了纯源码切换窥探按钮。
- **右栏（增强与分析层）**：
  - **AI增强信息概览**：发起独立的增强数据挂载请求。如果是完成态，将大语言模型生成的整体大纲（Overview）、散点关键盘点（Key Information）和抽取出的标签数组（Tags）呈现。如果 AI 端曾经处理出错，则允许人工介入点击带有动态遮罩的“重新生成”指令进行强行刷新。
  - **基于本文的实验分析链路**：单独再拉取本实体生命周期内参与过的所有独立“实验任务分析集”，供快速阅览每次具体运算的效果。


### 2.4 模型配置模块 (Model Configs)
**页面路由**：`/model-configs`

**1. 界面视图结构**
- **五彩参数科普卡片区**：为降低系统的模型配置门槛，页面顶部配置了 5 张知识提示卡。直观释义 Temperature（发散程度 0-2）、Top P（核概率覆盖度 0-1）、Top K、Max Tokens 和 Seed 的底层逻辑。
- **核心数据列表与表单配置**：表单提供配置名称与文本辅助说明维度、配置的实际物理底层模型唯一标识（`modelId`），并下挂那 5 大数值参数项。在数据载入和输入过程中采用了原生的 `number` 类型和阶梯限制 `step` 进行基础的数据越界遏制。


### 2.5 提示词模板管理模块 (Prompts)
**页面路由**：`/prompts` (列表页) 和 `/prompts/:id` (详情版本页)

#### 2.5.1 提示词列表页
旨在管理给 LLM 输出的所有可动态插补的抽象提示文本骨架。
- 页面首部同样有指导操作的卡片集，指出了当前系统在模板渲染时支持捕获并注入的上下文动态量（例如：`{content}`, `{author}`, `{title}`，运行时会被真文章数据替代）。
- 创建层面的功能非常轻化，只要一个总命名就可以创建出一个空的模板“树干”。随后所有具体的业务必须转场进入针对该模版的具体“控制树”版本详情页再行操作。

#### 2.5.2 提示词模板详情版控页 (PromptDetail)
当前系统抛弃了覆盖替换思路，采用了**不可变数据追加记录**的版本策略来管控实验的基线控制。
**核心交互与逻辑限制**：
- **版本切换与浏览**：页面内部使用了一个 `Select` 下拉加载出当前模板拥有的一系列递增迭代历史版本。
- **锁定与修改的隔离池**：
  - 假如通过选择器挑出了处于 `!immutable` 状态的版本，则允许在文本域中对实际 prompt 字符段进行任意键入和修改，并提供"保存"覆盖当前草稿的能力。此阶段可反复摩擦修改。
  - 如果最终确认了某版文案，需要显式点击“锁定版本”。之后该层级将发生冻结合约，所有文本域变为禁止写入状态（`immutable`），按钮组隐身消失。
- **继承下发机制**：如果需要继续做 prompt 分支演进，只有在上一个最高版本被“锁定防御”后，系统才会大方放开顶部悬浮的“创建新版本”控制流按钮。该行为触发后将为该模版的树中插入增量 `+1` 的全新未锁死叶子节点，并允许从头开始调校。


### 2.6 实验管理模块 (Experiments)
**页面路由**：`/experiments` (列表页) 和 `/experiments/:id` (详情页)

#### 2.6.1 实验列表页
**1. 特色向导区与状态约束**
- 在视图呈现的顶部，利用 5 个递进卡片将系统的终极核心能力操作进行了流程化剖分：“选择文章 -> 配置模型 -> 选择提示词 -> 启动分析 -> 查看结果反馈”。
- 实验列表中对状态流转极其敏感：对于仍在“RUNNING”批处理生命周期中的对象群，采取了前端禁用穿透、禁制编辑和防删除防御策略，只支持纯展示占位。

**2. “新增实验”复合组合流构建**
实验实质就是向后端发送一个组合处理的宏任务，该组合需涉及三大维度并全部在 `Dialog` 的单页复合链路中解决：
1. **文章圈入器（嵌套 Dialog 弹出式交互）**：由于批量实验需要圈定多条，使用了一个嵌套式搜索面板。用户从内层表格面板根据分类和分面搜索海量选中实体（由底层数据主键驱动），系统前台抽象出一个动态的 `Map` 池，并将这簇最终确认集通过快速删除 `Tag` 横向贴片的方式悬浮在主表单层上。
2. **选择模型配置**。
3. **安全提示词校验下发**：用户首先锁定了一个“模板”，前端通过挂接侦听器 `watch` 这项变动后，会去发包扫描拉取此模板的树。前端将自行将不安全、还未锁上（`immutable: false`）的不稳定草稿统统过滤清洗拦截掉，仅暴露可靠定型的全锁定提示词分支（v1，v2...），由实施人员选中唯一一条完成关联闭合，执行调度！


### 2.7 系统全局底层配置 (Settings)
**页面路由**：`/settings`

**1. 业务结构分割**
用于统揽全站系统级的底层参数下发，表单依照系统结构分割为 5 大模块：
- **采集器配置**：配置轮询触发网关和间隔，允许填入用以清洗脏数据的标题过滤词（`collector_title_filter_words` 面板）。
- **LLM/向量化 拓扑**：设定主调用基座的大模型路由出口 URL、秘钥及子切分使用的不同业务模型 ID（可以分别为推理语言和切片向量模型指向不同的接口通道）。
- **增强配置**：全局系统自研摘要的通用系统模板层预设内容配置和单体作业的局部并发限制量。
- **后台运维**：管理员级别的重置门槛策略密码及名号。
- **热点趋势**：例如特定聚合能力的统计分析延迟周期。

**2. 核心架构与持久流向封装**
- 对于不同输入形式（数值和字符串）底层建立了一套 `coerceValue` 类型强转规范及 `normalizeKey` 下划线转换格式处理器机制。不论是在拉取表征（`GET` 数组项）再散列为响应式对象属性去给 v-model 显示，亦或是重封装重构（`handleSave` 展开回实体序列）传递到持久层（`updateSettings`）过程中，都保持了强类型的可靠传递。
- 特立独行配备了“重置配置”拦截保护，一旦当前修改的属性有误，可点击重置直接利用组件状态快照进行回滚式渲染恢复。
